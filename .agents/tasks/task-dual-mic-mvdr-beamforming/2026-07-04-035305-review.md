# Dual-microphone MVDR beamformer with full Dart bridge

A header-only MVDR (Minimum Variance Distortionless Response) beamformer is added to the native DSP chain, positioned before the DNN denoiser. The audio engine conditionally opens a stereo input stream via Oboe, deinterleaves in the real-time callback, feeds both channels through the beamformer, and outputs mono. A JNI/Kotlin/Dart bridge exposes runtime enable/disable and status query all the way to Flutter. The change is retrocompatible: `beamformingEnabled` defaults to `false`, stereo never opens unless explicitly requested, and fallback to mono is handled when the device doesn't provide two channels.

**Watch for:** Overlap-add output buffer produces silence on first activation until the internal STFT buffer fills (confirmed). Rnn diagonal loading accumulates per-update, biasing the noise covariance estimate (confirmed). Frames beyond `kMaxBeamBlockSize` are silently dropped in the stereo path, leaving uninitialized output (confirmed). The Hann analysis+synthesis window pair introduces a ~0.75 gain scaling that is never compensated (likely).

## High-level view

The beamformer is a 256-point STFT overlap-add processor with 50% hop, performing per-bin MVDR weight computation using an exponentially-smoothed 2x2 noise covariance matrix. The steering vector is fixed at 0 degrees (frontal), and the VAD from the existing SceneAnalyzer gates Rnn updates. The overlap-add buffering logic introduces a startup latency gap where the output buffer hasn't accumulated enough data to emit valid samples, leading to an initial burst of zeros on first activation.

The `kMaxBeamBlockSize = 1024` cap in AudioEngine means any callback delivering more than 1024 frames will only process the first 1024 in stereo mode, leaving trailing output samples unwritten. Oboe's LowLatency mode typically delivers 128-512 frames, but no hard guarantee exists.

The runtime `setBeamformingEnabled` Dart API only toggles the atomic flag in the beamformer—it does not restart the stream in stereo. If the engine started with `beamformingEnabled=false` (mono), calling `setBeamformingEnabled(true)` later will set the flag but the callback still takes the mono memcpy path because `stereoInputAvailable_` is false. Yet `getBeamformingActive()` will return `true`, misreporting status to the UI.

The Hann window applied at both analysis and synthesis with 50% overlap produces a constant-overlap-add factor of 0.75 (not 1.0), attenuating the beamformer output by ~2.5 dB. For a hearing aid with calibrated gain stages (WDRC, MPO), this uncompensated attenuation shifts the entire downstream gain structure.

<summary>Issues (6)</summary>

1. **Rnn diagonal loading drift** — `kReg` is added to the diagonal on every `updateRnn` call, accumulating via the exponential smoothing. Move the loading to `computeMvdrWeights` (add just before inversion) or apply as `alpha * rnn + (1-alpha) * xxH + kReg*I` so smoothing bounds the contribution.
2. **Overlap-add startup silence** — On first enable, `outputBuf_` is zeroed and no processFrame has fired yet, so the first `numFrames` extracted will be zeros. Add a priming mechanism or passthrough ch0 until the first full STFT frame has been synthesized.
3. **Frame truncation at kMaxBeamBlockSize** — When `numFrames > 1024`, the stereo paths only process 1024 frames. The remainder of `outPtr` is uninitialized. Either loop over chunks or clamp `numFrames` with a log warning.
4. **OLA gain mismatch** — Hann window applied twice with 50% overlap sums to 0.75, not 1.0. Output is attenuated by ~2.5 dB. Apply `1.0f / 0.75f` normalization in `processFrame` or use sqrt-Hann.
5. **Runtime enable without stereo stream** — `setBeamformingEnabled(true)` from Dart sets the atomic flag but cannot retroactively open a stereo stream. `getBeamformingActive()` returns `true` even though mono data flows. Return an error or check `stereoInputAvailable_` in the getter.
6. **No test coverage** — No unit tests or integration tests for the MVDR module. The spec mentions Phase 5 validation but nothing is present in this diff.

## Overlap-add buffer synchronization gap

The `process()` method fills `outputBuf_` as STFT frames complete, then immediately reads `numFrames` samples from it. On the very first call, `inputBufPos_` starts at 0 and the buffer needs `kFftSize = 256` samples before `processFrame` fires. If the first callback delivers 128 frames (a typical Oboe burst at low latency), no frame will process and the output will be 128 zeros.

```cpp
// First call with numFrames=128:
// inputBufPos_ goes from 0 to 128 — no processFrame fires
// output[0..127] = outputBuf_[0..127] = all zeros
```

This manifests as a click or dropout on beamformer activation. For a hearing aid, even momentary silence is perceptible and disorienting.

## Rnn diagonal loading accumulates per-frame

In `updateRnn`, after the exponential smoothing step, `kReg` is unconditionally added to the diagonal:

```cpp
rnn_[k][0][0] += Complex(kReg, 0.0f);
rnn_[k][1][1] += Complex(kReg, 0.0f);
```

With `alpha = 0.98`, the steady-state contribution converges to `kReg / (1 - alpha) = 5e-5` per diagonal element—small at the current `kReg = 1e-6`, but conceptually wrong. Standard practice applies diagonal loading at inversion time, not during estimation. If `kReg` is ever increased for robustness on noisy devices, this accumulation would meaningfully bias the MVDR weights toward delay-and-sum behavior, negating the spatial filtering benefit.

## Frame truncation beyond kMaxBeamBlockSize

```cpp
int framesToProcess = std::min(numFrames, kMaxBeamBlockSize);
for (int i = 0; i < framesToProcess; ++i) {
    beamCh0_[i] = inPtr[i * 2];
    beamCh1_[i] = inPtr[i * 2 + 1];
}
```

If `numFrames > 1024`, frames 1024 onward in `outPtr` are never written. The trailing portion will contain stale data from the output stream buffer. The downstream DSP pipeline processes this as if it were valid audio. Oboe's `LowLatency` mode typically stays under 512 frames, but the contract doesn't guarantee it. A loop processing in `kMaxBeamBlockSize` chunks fixes this without increasing buffer sizes.

## Hann window COLA normalization

```cpp
// In processFrame:
outputBuf_[n] += frameBuf[n] * window_[n];
// Comment says "Hann 50% overlap: factor ya implicito" — but it's 0.75, not 1.0
```

The output is consistently 2.5 dB quieter than the input. The DNN denoiser and WDRC after the beamformer will see a signal below its calibrated level, potentially causing the WDRC to apply more gain than intended to compensate.

## Runtime enable semantics mismatch

The Dart API exposes `setBeamformingEnabled(bool)` as a live toggle, but the audio stream channel count is determined at `start()` time. If beamforming is enabled post-start on a mono stream, `stereoInputAvailable_` remains false and the callback takes the mono memcpy path. The beamformer never receives stereo data.

The JNI layer silently accepts the call. `isBeamformingActive()` reads only the atomic flag, not the stream state, so `getBeamformingActive()` returns `true` even when no beamforming is occurring. This reports false status to the UI.

## SceneAnalyzer VAD is one block stale

The beamformer reads `sceneAnalyzer_.getVad().isVoiceActive()` before SceneAnalyzer processes the current frame (SceneAnalyzer runs after the beamformer in the callback). The VAD state is therefore one callback-block old. At typical 128-frame bursts (2.7ms at 48kHz), this is well within acceptable latency for Rnn gating and consistent with the referenced papers' real-time MVDR implementations. The consequence: the first noise frame after voice will not update Rnn, and the first voice frame after noise will briefly contaminate Rnn with one frame of speech. Neither is perceptually significant given the 0.98 smoothing factor.
