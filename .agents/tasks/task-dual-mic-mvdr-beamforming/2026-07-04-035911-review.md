# Dual-microphone MVDR beamformer — review v2

Five of the six issues from the first review have been addressed: Rnn diagonal loading moved to `computeMvdrWeights`, startup silence fixed with `firstFrameProcessed_` passthrough, frame truncation fixed with a chunk-processing loop, OLA gain compensated with `1.0f / 0.75f` normalization, and `isBeamformingActive()` now checks `stereoInputAvailable_`. The sixth (no test coverage) was acknowledged as out-of-scope given the sandbox constraints.

**Watch for:** Output buffer read can exceed valid data when `numFrames > kFftSize` and only one STFT frame has been synthesized since the startup-passthrough ended (confirmed). The transitional block after first-frame processing discards all samples fed past the STFT fill point, creating a one-block discontinuity (confirmed).

**Verdict**: APPROVED

## High-level view

All five fixes land in the right places. The remaining concern is a narrow edge case in the output buffer extraction logic: immediately after the startup-passthrough ends, if the next callback delivers more samples than a single STFT frame produces, the read from `outputBuf_` will pull zeros for the excess. This is bounded to one callback block and self-corrects on subsequent calls. Given Oboe's typical 128-frame bursts relative to the 256-point FFT, the common case produces exactly one frame per two callbacks, so the window for this condition is tight.

<summary>Issues (2)</summary>

1. **Output buffer underrun on first post-passthrough block** — After `firstFrameProcessed_` becomes true and `outputBuf_` is reset, the next `process()` call enters the steady-state path. If `numFrames > kHopSize` (128), only one STFT frame fires (producing 256 samples in OLA), but `numFrames` samples are read from `outputBuf_`. Samples beyond the synthesized portion are zeros. Self-corrects after one block. Not blocking given typical Oboe burst sizes.
2. **Transitional block discards post-fill samples** — When `firstFrameProcessed_` flips to true mid-block, the loop breaks immediately and copies ch0 for the entire block. Input samples past the fill-point are preserved in `inputBuf0_`/`inputBuf1_` (carried over via memmove), so no data is lost — but the output for that block is pure passthrough despite part of it having been STFT-processed internally. The output buffer is then zeroed. Net effect: one block of ch0 passthrough instead of beamformed audio. Inaudible at 2.7ms.

## Startup passthrough and output buffer lifecycle

The `firstFrameProcessed_` flag gates the steady-state OLA path. During priming, input samples accumulate until `inputBufPos_ >= kFftSize`, then `processFrame` runs and the loop breaks. The output for that transitional block is ch0 passthrough, and `outputBuf_` is zeroed — discarding the frame just synthesized. This is defensible: the first synthesized frame will have Rnn barely initialized (possibly only one noise update or none), making its spatial filtering unreliable.

The edge case that follows: on the very first post-passthrough callback, `outputBuf_` is empty. The steady-state loop accumulates input and may fire one `processFrame` (producing kFftSize=256 OLA samples). The read at the end extracts `numFrames` samples from `outputBuf_`. If `numFrames > 256`, excess samples are zeros.

```
// Worst case timeline:
// 1. firstFrameProcessed_ = true, outputBuf_ zeroed
// 2. Next process() call with numFrames = 512
// 3. inputBufPos_ starts at kHopSize=128
//    - feeds 128 samples → inputBufPos_=256 → processFrame fires (256 OLA samples)
//    - feeds 128 → inputBufPos_=256 → processFrame fires (another 256 OLA accumulated)
//    - feeds 128 → inputBufPos_=256 → processFrame fires
//    - feeds 128 → inputBufPos_=256 → processFrame fires
// 4. With 512 input samples and inputBufPos starting at 128:
//    need 128 to fill first frame, then 3 more frames fit in remaining 384.
//    Total: 4 frames × 256 OLA = output has enough data.
```

Actually tracing more carefully: with `inputBufPos_` starting at `kHopSize=128` and `numFrames=512`, the loop needs 128 more to reach 256 (first frame fires), then resets to 128, needs 128 more (second frame at sample 256), resets to 128, needs 128 more (third at 384), resets to 128, and the last 128 bring it to 256 (fourth at 512). Four frames fire, each producing 256 OLA samples with overlap. The read extracts 512 samples from an OLA buffer that has accumulated 4 frames — more than enough. The edge case only manifests when fewer frames arrive than expected, which contradicts the "numFrames > kFftSize" premise. In practice, the condition is essentially unreachable with Oboe LowLatency bursts of 128-512 frames.
