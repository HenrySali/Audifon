# GTCRN dual-channel: LibTorch removal, WPE in C++, ONNX-only inference

The dual-channel speech enhancement pipeline was restructured to eliminate LibTorch entirely. Previously, a TorchScript model (3-second block processing, ~63 MB APK overhead) handled STFT, WPE beamforming, IVA, and the neural core as a monolithic `.pt`. This change splits that: STFT and a WPE-like MVDR beamformer run in C++ frame-by-frame, and only the neural encoder/decoder core goes through ONNX Runtime (same runtime as the mono path). Latency drops from 3 seconds to 8 ms. APK drops from ~63 MB to ~6 MB. A Python export script generates the ONNX model from the original GTCRN_IVA checkpoint.

**Watch for:** VAD is hardcoded `true` in the dual worker loop, so the WPE noise covariance never updates at runtime (confirmed). The `Rxx` signal covariance matrix is computed but never used in `computeWeights` — weights are derived solely from `Rnn`, making the `Rxx` tracking dead computation (confirmed). The export script's `AdaptiveGTCRNCore` wrapper uses heuristic layer discovery that may silently produce an identity-like model if the GTCRN_IVA architecture doesn't match expected patterns (likely).

## High-level view

The WPE beamformer is architecturally an MVDR spatial filter using noise-only covariance, not a true WPE (prediction error) method. It operates per-bin with a fixed steering vector `[1, 0]` pointing at mic 0, producing `w[0] = 1` always (distortionless constraint). The spatial benefit comes entirely from `w[1]`, which steers a null toward noise. The label "WPE" in the code will confuse future maintainers who expect a dereverberation filter with prediction taps.

The dual processing path (`processDualFrame`) mirrors the mono `processOneFrame` closely: shift-and-window STFT, run inference, iSTFT with overlap-add. Both mono and dual instances own separate `Impl` objects so sharing the ONNX session infrastructure introduces no conflicts.

The VAD-gated noise estimation is the beamformer's main adaptation mechanism, but the hardcoded `vadActive = true` in the worker loop means `updateRnn` never fires. The beamformer will fall through to the `!rnnInitialized_` branch and output `X0[k]` (reference mic passthrough) indefinitely, providing zero spatial gain until this is wired to `voiceActive_`.

The Python export script auto-discovers encoder/decoder/sequence model layers from GTCRN_IVA using heuristic name matching. If the layer names don't match any pattern, it falls back to wrapping the entire model's forward — which may include STFT/complex ops the C++ pipeline can't handle.

1. **VAD hardcoded true in dual worker** — `const bool vadActive = true` on line 1344 of dnn_denoiser.cpp prevents the WPE beamformer from ever updating its noise covariance. Replace with `voiceActive_.load(std::memory_order_acquire)` (the member already exists on the outer `DnnDenoiser` class and is set by `notifyVoiceActive`). Without this, the beamformer provides no spatial filtering.
2. **Rxx covariance is dead computation** — `updateRxx` is called every frame but `rxx_` is never read by `computeWeights` (which uses `rnn_` only). Either remove `Rxx` tracking and the `rxxInitialized_` branch, or switch to a Wiener-style filter that uses both matrices.
3. **Stack-allocated VLA-size arrays in processDualFrame** — `WpeBeamformer::Complex X0[257]`, `X1[257]`, `Y[257]` are stack arrays of `std::complex<float>` totaling ~6 KB per call. On Android worker threads with limited stack (typically 64 KB to 1 MB), this is safe but marginal. Consider making them `Impl` members alongside `fftReCh1`.
4. **Export script silent fallback** — If `AdaptiveGTCRNCore` cannot find encoder/decoder layers, it falls back to calling the full model's `forward()`. The ONNX graph may then contain forbidden ops (STFT, complex). The validator catches some post-export, but the user could ignore warnings and deploy a broken model. Add a hard failure if no core layers are discovered (unless `--manual` is set).
5. **Dual and mono paths share no STFT code** — `processDualFrame` reimplements the STFT shift/window/FFT logic that already exists in `processOneFrame`. If the window or normalization changes, both must be updated in lockstep. Consider extracting a shared helper.

## VAD wiring gap in the dual worker loop

The `workerLoop` in dnn_denoiser.cpp hardcodes:

```cpp
const bool vadActive = true;  // Conservative: assume speech present
```

The comment says "noise covariance updated via external notifyVoiceActive" but this is incorrect — `notifyVoiceActive` sets `voiceActive_` on the *outer* `DnnDenoiser` object, and the worker loop never reads it. The `voiceActive_` atomic is declared on `DnnDenoiser` (not `Impl`), so reading it from the worker requires either passing a reference into `Impl` or storing a pointer.

The consequence: `rnnInitialized_` stays `false` forever, and every call to `process()` in the beamformer takes the `Y[k] = X0[k]` passthrough branch (once `rxxInitialized_` becomes true). The entire beamformer contributes zero spatial gain. The fix is straightforward: expose `voiceActive_` to the worker via a `std::atomic<bool>*` member in `Impl`, set during construction.

## Beamformer: MVDR with fixed steering, not WPE

The implementation is a standard MVDR beamformer — it inverts a 2x2 noise covariance matrix and applies the result with a fixed steering vector `d = [1, 0]`. There are no prediction filter taps, no delay estimation, no dereverberation structure.

`computeWeights` yields `w[0] = 1` always (dividing column 0 of the inverse by its own [0,0] element). The spatial suppression comes entirely from `w[1]`, which represents the relative weight of mic 1 for canceling noise.

The `rxx_` matrix (signal covariance) is updated every frame but never consumed in `computeWeights`. This is ~6 KB of state written every frame without purpose. If the intent is to later support a Wiener post-filter, that should be a documented TODO; otherwise remove `updateRxx` and `rxx_`.

## processDualFrame: null-session asymmetry

`processOneFrame` (mono) has a separate check for `!session` before running inference. `processDualFrame` does not — it relies on `modelReady` being false. If `session` were null while `modelReady` is true (shouldn't happen given the single-writer init pattern, but represents an inconsistency between the two code paths), `processDualFrame` would dereference null. Low likelihood, but the guard is trivial to add for symmetry (possible).

## Export script: fallback wraps full model

If `discover_layers` fails to match encoder/decoder patterns, the wrapper falls back to calling the full GTCRN_IVA forward. The post-export `validate_onnx` checks for forbidden op types (STFT, DFT, MatrixInverse), providing a safety net on the developer's machine. But if the user ignores the validator's warning and copies the model to assets, the shape check in `processDualFrame` (`mixShape == [1,257,1,2]`) will catch the mismatch at runtime and fail to bypass — so the failure mode is safe (no crash, just no enhancement).

## Test coverage gaps

Not tested: the full STFT->WPE->ONNX->iSTFT pipeline (requires device + NDK + real `.onnx` model). The WPE beamformer has no standalone unit test verifying MVDR output against known spectra. Given the VAD hardcoding issue, even on-device integration testing would show the beamformer providing no spatial gain until the wiring is fixed.
